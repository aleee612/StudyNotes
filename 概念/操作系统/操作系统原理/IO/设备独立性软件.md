### 设备独立性软件定义
- 又称与设备无关的软件
- 核心作用：实现各类性质的公共操作，为上层提供统一、抽象的I/O接口
1. 提供统一的驱动程序接口
2. 缓冲管理
3. 差错控制h
4. 独占设备的分配与回收
5. 提供统一的逻辑数据块

### 缓冲区
#### 磁盘高速缓存
| **特性**       | **CPU 硬件缓存 (CPU Cache)** | **磁盘高速缓存 (Disk Cache)**    |
| ------------ | ------------------------ | -------------------------- |
| **解决谁和谁的矛盾** | CPU 速度快 vs 内存速度慢         | 内存速度快 vs 磁盘/固态速度慢          |
| **物理位置**     | 属于 CPU 芯片的一部分（硬件）        | 属于 **内存 (RAM)** 的一部分（软件管理） |
| **缓存的内容**    | 内存中数据的副本                 | 磁盘中盘块数据的副本                 |
| **管理方式**     | 纯硬件自动管理                  | 操作系统 (OS) 负责管理             |
#### 缓冲区
| **缓冲类型** | **工作原理与特点**                                          | **每块数据的平均处理时间（公式）**                                               | **优点**                                         | **缺点 / 适用场景**                         | 图示                                   |
| -------- | ---------------------------------------------------- | ----------------------------------------------------------------- | ---------------------------------------------- | ------------------------------------- | ------------------------------------ |
| **单缓冲**  | 操作系统在内存中只分配**一个**缓冲区。<br>设备和CPU**只能串行/交替**使用它。       | $$\max(T, C) + M$$                                                | 结构最简单，消耗内存空间最少。                                | 无法做到完全的并行，设备和CPU之间依然存在明显的等待。          | ![[Pasted image 20260519104543.png]] |
| **双缓冲**  | 操作系统分配**两个**缓冲区（1号和2号）。设备向1号写时，CPU可以从2号读，实现**交替并行**。 | $$\max(T, C+M)$$<br><br>  <br><br>_(通常 $M$ 很小，常近似为 $\max(T, C)$)_ | 实现了输入和处理的**并行**。如果 $T \approx C$，设备和CPU几乎不用等待。 | 缓冲区利用率依然有限。如果输入和处理速度相差悬殊，仍会有缓冲区闲置或等待。 | ![[Pasted image 20260519104552.png]] |
| **循环缓冲** | 将**多个**固定大小的缓冲区连接成一个**循环队列**。<br>包含输入缓冲区、满缓冲区、空缓冲区等。 | 依赖于队列长度和速度匹配，趋近于<br><br>$$\max(T, C)$$                            | 适合**输入和输出速度基本匹配，但有局部波动**的连续高并发场景。              | 缓冲区个数固定，缺乏弹性。属于**专用**缓冲，多个进程无法高效共享。   | ![[Pasted image 20260519104604.png]] |
| **缓冲池**  | 属于**全局共享**资源。<br>由多个缓冲区组成，设置三个队列（空、输入、输出）和四种工作缓冲区。   | 按需动态分配，高效趋近系统极限<br><br>$$\max(T, C)$$                             | **灵活性最高**。<br>实现了多进程共享，大幅提高了内存利用率和系统吞吐量。       | 结构最复杂，系统设计和管理的开销相对较大。                 | ![[Pasted image 20260519104611.png]] |

### 设备分配与回收
#### 控制表
| **数据结构名称** | **英文简称** | **核心作用**                              | **包含的主要字段/信息**                                                               | **数量关系** |
| ---------- | -------- | ------------------------------------- | ---------------------------------------------------------------------------- | -------- |
| **设备控制表**  | **DCT**  | **每个物理设备**对应一张表，用于记录该设备的具体状态和控制信息。    | • 设备类型、设备标识符<br>• 设备状态（忙/闲）<br>• 与该设备连接的 **COCT** 指针<br>• 进程等待队列指针（等待该设备的进程） | 每个物理设备一张 |
| **控制器控制表** | **COCT** | **每个设备控制器**对应一张表，负责管理和控制具体的一组设备。      | • 控制器标识符<br>• 控制器状态（忙/闲）<br>• 与该控制器连接的 **CHCT** 指针<br>• 控制器队列指针（等待该控制器的进程）   | 每个控制器一张  |
| **通道控制表**  | **CHCT** | **每个通道**对应一张表（现代系统或大型机中），用于管理通道状态。    | • 通道标识符<br>• 通道状态（忙/闲）<br>• 通道队列指针（等待该通道的进程）                                 | 每个通道一张   |
| **系统设备表**  | **SDT**  | **整个系统仅此一张**，记录系统内所有物理设备的状态，是设备分配的入口。 | • 设备类型、设备标识符<br>• 指向对应设备 **DCT** 的指针<br>• 设备驱动程序入口                           | 整个系统仅一张  |
#### 分配算法
| **算法名称**              | **分配策略/核心思想**                                      | **优点**                                     | **缺点**                                       | **适用场景**                              |
| --------------------- | -------------------------------------------------- | ------------------------------------------ | -------------------------------------------- | ------------------------------------- |
| **先来先服务算法**<br>(FCFS) | 根据进程**请求设备的先后顺序**进行分配。系统维护一个设备请求队列，新请求排在队尾，分配时取队头。 | • 算法简单、易于实现<br>• 具有极高的**公平性**，不会出现进程“饿死”现象 | • 效率较低<br>• 未考虑进程的紧迫程度（紧急任务需要排队）             | 适用于对实时性要求不高、设备请求较均匀的普通通用系统。           |
| **优先级高者优先算法**(HPF)    | 根据进程的**优先级高低**进行分配。系统将设备优先分配给当前请求队列中优先级最高的进程。      | • 灵活性高<br>• 能确保**紧急、重要的任务**优先获得设备并执行       | • 可能会导致低优先级进程长期拿不到设备，产生**饿死** 现象<br>• 实现相对复杂 | 适用于对实时性要求高、有紧急任务处理需求的系统（如嵌入式、实时操作系统）。 |
#### 设备分配的步骤
> 查设备 -> 查控制器 -> 查通道 -> 成功启动
1. 第一步：分配设备（查 SDT 和 DCT）
	- **操作：** 进程提出 I/O 请求后，系统首先根据设备物理名查找**系统设备表 (SDT)**，找到该设备的**设备控制表 (DCT)**。
	- **验证：** 检查 DCT 中的设备状态。
    - 如果设备**忙**，则将进程的 PCB 挂到该设备的**等待队列**中；
    - 如果设备**闲**，系统会根据算法（如 FCFS 或优先级）将设备**分配**给该进程。
2. 第二步：分配控制器（查 COCT）
	- **操作：** 设备分配成功后，系统通过 DCT 中的指针，找到与该设备连接的控制器的**控制器控制表 (COCT)**
	- **验证：** 检查 COCT 中的控制器状态。
    - 如果控制器**忙**，则将进程挂到该控制器的**等待队列**中；
    - 如果控制器**闲**，系统将该控制器**分配**给该进程。
3. 第三步：分配通道（查 CHCT）
	- **操作：** 控制器分配成功后，系统通过 COCT 中的指针，找到与该控制器连接的通道的**通道控制表 (CHCT)**
	- **验证：** 检查 CHCT 中的通道状态
    - 如果通道**忙**，则将进程挂到该通道的**等待队列**中；
    - 如果通道**闲**，系统将该通道**分配**给该进程。
4. 第四步：启动 I/O 设备进行数据传输
	- **操作：** 只有当**设备、控制器、通道**三者都成功分配给该进程后，这次设备分配才算真正完成。
    - **结果：** 系统正式启动 I/O 设备，开始为该进程进行数据传输和输入输出操作。
#### LUT
|**考点分类**|**核心细节 / 考点内容**|**考点说明与理解**|
|---|---|---|
|**核心定义**|记录进程使用的**逻辑设备名**与系统分配的**物理设备名**之间的映射关系表。|用于建立两者的对应关系。|
|**引入目的**|主要是为了实现**设备独立性**（用户编程时无需关心具体物理设备）。|更换物理设备时不需要修改应用程序。|
|**表项内容**|包含三个核心字段：**逻辑设备名**、**物理设备名**、**设备驱动程序入口地址**。|通过驱动程序入口直接调用该设备的驱动代码。|
|**查表时机**|当进程使用**逻辑设备名**提出 I/O 请求时，系统自动查找该表。|将逻辑名转换为物理名，从而找到对应的 DCT。|
|**单用户设置**|**整个系统仅设置一张 LUT**。|缺点：不允许不同用户使用相同的逻辑设备名（易冲突）。|
|**多用户设置**|**为每个用户（或进程）配置一张 LUT**。|优点：不同用户可使用相同逻辑名（如都叫 `/dev/printer`），互不干扰。|
|**存储位置**|多用户环境下，LUT 通常存储在用户进程的 **PCB（进程控制块）** 中。|随着进程切换而切换。|
|**动态建立**|进程在**打开设备/申请设备**时动态向 LUT 写入新表项。|设备分配成功后才真正建立映射。|
|**动态销毁**|进程在**关闭设备/释放设备**时释放该表项。|退出时擦除，以便其他进程复用。|

### SPOOLing
| **组成部分**        | **存储位置**    | **核心作用 / 职责**                                                                                                  | **角色比喻**  |
| --------------- | ----------- | -------------------------------------------------------------------------------------------------------------- | --------- |
| **输入井和输出井**     | **磁盘**（外存）  | 在磁盘上开辟的高速暂存区<br>• **输入井**：暂存从输入设备输入的数据<br>• **输出井**：暂存用户程序要求输出的数据                                              | 数据大仓库     |
| **输入缓冲区和输出缓冲区** | **内存**      | 在内存中开辟的临时过渡区<br>• **输入缓冲区**：暂存由输入设备传送来的数据，随后再转存到输入井<br>• **输出缓冲区**：暂存从输出井传送来的数据，随后再传送到输出设备                     | 运输中转站     |
| **输入进程和输出进程**   | **内存**（运行中） | 负责数据传输的“搬运工”<br>**输入进程（预输入程序）**：模拟脱机输入，利用 CPU 空闲将数据从输入设备搬到输入井<br>**输出进程（缓输出程序）**：模拟脱机输出，利用 CPU 空闲将数据从输出井搬到输出设备 | 搬运工 / 快递员 |
| **井管理程序**       | **操作系统内核**  | 负责控制和管理输入井、输出井的组织、分配以及数据的读写                                                                                    | 仓库管理员     |
