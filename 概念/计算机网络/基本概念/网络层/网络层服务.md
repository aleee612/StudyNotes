- 在网络层，提供的为**主机**之间的逻辑通信
### 尽力而为服务
- 不保证传送的分组以其发送的顺序被接收
- 不保证分组最终交付
- 不保证端到端时延RTT
- 不保证有最小带宽

### SDN
#### 数据平面（硬件）
##### 主要功能
**转发：**
- 当一个分组到达某路由器的一条输入链路时，该路由器必须将该分组移动到适当的输出链路
- 时间尺度短，通常通过硬件实现
**IP：网际协议**
- 服务模型：尽力而为交付服务
	- 但并不做任何确保
- 不可靠服务
- 每台主机有一个IP地址
**转发表**：
- 检测到达分组首部 -> 用这些首部值在路由器转发表中索引
#### 控制平面（软件）
##### 主要功能
**路由选择：**
- 当分组从发送方流向接收方时，网络层必须决定这些分组采用的路由或路径
- 路由选择算法：计算该去往路径的方法
[[路由选择算法]] 
#### 传统
- 假想：所有都为操作员配置，不需要任何路由选择协议
#### SDN：软件定义网络
- 控制平面路由选择功能与物理路由器分离，路由选择设备仅执行转发
- 远程控制器（选择功能）计算并分发表，实现在远程数据中心中由ISP / 第三方管理
- 体现网络是“软件定义”的
#### 四种接口
| **接口名称** | **面向对象 / 连接位置**  | **主要功能与作用**                          | **典型协议/特点**       |
| -------- | ---------------- | ------------------------------------ | ----------------- |
| 北向接口     | **上层应用 <-> 控制器** | 向应用层提供标准化API，屏蔽底层细节，支持开发者构建定制化网络应用。  | 灵活、可编程性强          |
| 南向接口     | **控制器 <-> 转发设备** | 实现双向通信，由控制器统一管理异构硬件，将逻辑动态下发至数据平面。    | **OpenFlow**、P4 等 |
| 东西向接口    | **控制器 <-> 控制器**  | 用于控制器集群内部通信，提升控制平面的**可靠性**与**可拓展性**。 | 集群同步、扩展性          |

### 提供的两种服务
|               | **数据报服务**              | **虚电路服务**                          |
| ------------- | ---------------------- | ---------------------------------- |
| **连接的建立**     | 不需要                    | 必须**有**                            |
| **目的地址**      | 每个分组都有完整的目的地址          | 仅在建立连接阶段使用，之后每个分组使用长度较短的虚电路号（VCID） |
| **路由选择**      | 每个分组独立地进行路由选择和转发       | 属于同一条虚电路的分组按照同一路由转发                |
| **分组顺序**      | 不保证分组的有序到达             | 保证分组的**有序**到达                      |
| **可靠性**       | **不保证可靠通信**，可靠性由用户来保证  | 可靠性由网络自身来保证（即**具有可靠性**）            |
| **对网络故障的适应性** | 可寻找新的路径转发分组，对故障的适应能力强  | 所有经过故障节点的虚电路均不能正常工作                |
| **差错处理和流量控制** | 由用户主机进行流量控制，不保证数据报的可靠性 | 可由分组交换网负责，也可由用户主机负责                |

| **特性**   | PVC                   | SVC                        |
| -------- | --------------------- | -------------------------- |
| **中文名称** | **永久虚电路**             | **交换虚电路**                  |
| **建立方式** | 由网络管理员通过管理系统**手动配置**。 | 由端系统通过信令协议**动态建立**。        |
| **存在时间** | 长期存在，即使没有数据传输也不会拆除。   | 临时存在，数据传输完后自动拆除。           |
| **连接过程** | 无需呼叫建立过程，直接传输数据。      | 类似于打电话：呼叫建立 -> 数据传输 -> 释放。 |
| **适用场景** | 两个固定地点之间的高频、大流量数据传输。  | 任意节点间的临时、低频通信。             |
| **优点**   | 效率高，省去了建立连接的信令开销。     | 灵活度高，节省网络地址/标识资源。          |

### 拥塞控制
| **对比维度** | **开环控制**                                | **闭环控制**                           |
| -------- | --------------------------------------- | ---------------------------------- |
| **核心思路** | **预防为主**<br>在系统设计时预先考虑拥塞因素，通过静态策略防止拥塞发生 | **反馈调节**<br>监测网络运行状态，动态检测并缓解已发生的拥塞 |
| **实时性**  | **不依赖**实时反馈，一旦运行，机制固定                   | **高度依赖**实时反馈机制，根据状态动态调整            |
| **典型手段** | 确定何时接收新流量、何时及丢弃哪些分组、确定调度策略等             | 监测拥塞位置与程度，反馈至控制节点，调整网络运行           |
| **控制特点** | 静态、预防性、不看当前网络状态                         | 动态、修正性、基于反馈环路                      |
