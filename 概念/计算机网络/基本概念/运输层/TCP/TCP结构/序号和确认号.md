- 作用：提供可靠数据传输
### 序号
![[Pasted image 20251013214004.png]]
- 建立在**字节流**而不是报文段序列上
- 一个报文段的序号 = **该报文段首字节的编号**
- TCP 对每一个字节都进行编号
- SYN=1 时，序号字段中的值为初始序号 ISN
- 消耗序号的情况
    - SYN 报文段消耗 1 个序号
    - FIN 报文段消耗 1 个序号
    - 普通数据按字节数消耗序号
- 序号字段长度为 **32 bit**，采用循环编号
    - 范围：
$$0\sim2^{32}-1$$
- 达到最大值后重新从 0 开始（序号绕回）

---
### 确认号
> **期望收到的下一个字节的序号**
> 即：
```text
确认号 = 已正确接收的最后一个字节序号 + 1
```
例如：
```text
收到 0~999 号字节
↓
ACK = 1000
```
- ACK=1000 表示：
    - 0~999 已经正确收到
    - 希望下一个收到 1000 号字节

---
### 累积确认
> TCP 只确认**按序到达且连续的一段数据**，因此 TCP 提供的是**累积确认**
#### 例
发送：
```text
1000~1999
2000~2999
3000~3999
```
若：
- 第 1 段收到
- 第 2 段丢失
- 第 3 段收到
则接收方仍返回：
```text
ACK = 2000
```
表示：
```text
1000~1999 已收到
2000 开始的数据还没连续收到
```
不会返回 ACK=4000
因此：
> TCP 只认第一个丢失字节之前的连续数据

---
### 捎带确认
TCP 是全双工通信
若接收方也有数据发送，可以：
- 在发送数据报文时顺便携带 ACK；
- 减少报文数量，提高效率

---
### 重复确认
当收到失序报文段时：
- 接收方立即发送重复 ACK
- 确认号保持不变
例如：
```text
收到：
1000~1999
3000~3999

缺少：
2000~2999
```
则连续发送：
```text
ACK=2000
ACK=2000
ACK=2000
```
发送方收到 **3 个重复 ACK** 后：
→ 触发快速重传

---
### 序号绕回
序号和确认号都是 32 位
```text
4294967294
4294967295
0
1
2
……
```
按模 $2^{32}$ 编号
利用：
- MSL（最大报文寿命）
- 时间戳 PAWS
避免新旧报文混淆

---
> 确认号确认的是字节，不是报文段
错误：
```text
ACK=3 表示确认第3个报文段
```
正确：
```text
ACK=3 表示 0~2 号字节已经收到
```
### 序号的长度
| 字段      | 长度     |
| ------- | ------ |
| 源端口     | 16 bit |
| 目的端口    | 16 bit |
| 序号 seq  | 32 bit |
| 确认号 ack | 32 bit |
| 窗口      | 16 bit |
