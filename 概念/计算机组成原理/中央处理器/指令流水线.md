### 五段流水结构
| 流水段 | 英文  | 功能          |
| --- | --- | ----------- |
| 取指  | IF  | 取指令、修改 PC   |
| 译码  | ID  | 分析指令、取寄存器   |
| 执行  | EX  | 算术逻辑运算、形成地址 |
| 访存  | MEM | 访问数据存储器     |
| 写回  | WB  | 结果写入寄存器     |
口诀：
```text
取 → 译 → 执 → 访 → 写
```

#### 顺序执行与流水执行比较
| 项目         | 顺序执行     | 流水执行       |     |
| ---------- | -------- | ---------- | --- |
| 并行性        | 无        | 有          |     |
| 单条指令时间     | $k·t$    | $k·t$      |     |
| 吞吐率        | 低        | 高          |     |
| 完成 n 条指令时间 | $nkt$    | $(k+n−1)t$ |     |
| 极限吞吐率      | $1/(kt)$ | $1/t$      |     |
| 极限加速比      | $1$      | $k$        |     |
> 流水线提高吞吐率，不缩短单条指令执行时间
#### 使用流水线的条件
##### 基本条件
| 条件          | 含义          |
| ----------- | ----------- |
| 指令能够分解成若干阶段 | 每个阶段完成不同功能  |
| 各阶段功能相对独立   | 前后阶段可并行工作   |
| 各段所需时间尽可能接近 | 避免某一段成为瓶颈   |
| 各阶段连续提供数据   | 保证流水线不断流    |
| 阶段之间有缓冲寄存器  | 保存中间结果，实现并行 |
##### 各流水段时间要求
| 情况      | 结果      |
| ------- | ------- |
| 各段时间相等  | 效率最高    |
| 各段时间接近  | 效率较高    |
| 某一段明显过长 | 吞吐率受限   |
| 某一段明显过短 | 硬件利用率降低 |
#### 目的
| 项目   | 内容           |
| ---- | ------------ |
| 根本目的 | 提高系统吞吐率      |
| 本质   | 时间并行技术       |
| 提高对象 | CPU利用率、硬件利用率 |
| 不提高  | 单条指令执行时间     |
| 不改变  | 单条指令的执行过程    |
#### 流水线重要公式
| 名称     | 公式                       | 极限情况        |
| ------ | ------------------------ | ----------- |
| 流水周期   | $t=max(各段时间)$            | ——          |
| 顺序执行时间 | $T_顺=n·k·t$              | ——          |
| 流水执行时间 | $T_流=(k+n−1)t$           | $n≫k_时≈nt$  |
| 实际吞吐率  | $T_P=n/T_流$              | ——          |
| 最大吞吐率  | $T_{Pmax}=1/t$           | ——          |
| 加速比    | $S=T_顺/T_流= n·k/(k+n−1)$ | $S_{max}=k$ |
| 效率     | $η=S/k=n/(k+n−1)$        | $η_{max}=1$ |

### 流水线冒险
| 类型   | 原因     | 例子           | 解决方法         |
| ---- | ------ | ------------ | ------------ |
| 结构冒险 | 争夺硬件资源 | IF、MEM同时访问主存 | 增加资源、暂停      |
| 数据冒险 | 数据尚未产生 | 后一条指令使用前一条结果 | 旁路、暂停、编译优化   |
| 控制冒险 | 转移地址未知 | if、jmp、beq   | 分支预测、暂停、延迟分支 |
口诀：
```text
结构抢资源
数据等结果
控制等方向
```

| 类型（中文） | 英文缩写    | 实际执行顺序            | 含义                             | 是否真相关     | 说明                                                       |
| ------ | ------- | ----------------- | ------------------------------ | --------- | -------------------------------------------------------- |
| 写后读    | **RAW** | **Write → Read**  | 前一条指令**写**某个寄存器，后一条指令**读**该寄存器 | 真相关       | 1. 后一条指令确实依赖前一条产生的数据<br>2. 经典五级流水线中最常见的数据冒险              |
| 读后写    | **WAR** | **Read → Write**  | 前一条指令**读**某个寄存器，后一条指令**写**该寄存器 | 假相关（反相关）  | 1. 两条指令没有数据传递，只是都访问同一个寄存器<br>2. 经典五级流水线一般不会发生，多见于乱序执行    |
| 写后写    | **WAW** | **Write → Write** | 两条指令都**写**同一个寄存器               | 假相关（输出相关） | 1. 两条指令没有数据传递，但必须保证最终写入顺序正确<br>2. 经典五级流水线一般不会发生，多见于乱序执行。 |
#### 数据冒险解决方法
| 方法       | 原理              | 是否停顿 |
| -------- | --------------- | ---- |
| 插入气泡     | 暂停流水线           | 是    |
| 数据旁路（转发） | EX/MEM结果直接送后续阶段 | 否    |
| 编译优化     | 调整指令顺序          | 一般否  |
#### 控制冒险解决方法
| 方法    | 原理         |
| ----- | ---------- |
| 暂停流水线 | 等待分支结果     |
| 分支预测  | 预测跳转方向     |
| 延迟分支  | 将无关指令放入延迟槽 |
> 若预测失败则 **Flush**（冲刷流水线）清除错误指令（即流水线被舍弃）

> [!warning]- Load-Use 数据冒险
> 1. 什么是 Load-Use 冒险？
>	- 在经典的五段流水线（IF, ID, EX, MEM, WB）中，**Load-Use 冒险**是一种特殊的数据冒险（RAW，写后读）
>	- 它发生在：前一条指令是**从内存中读数据（Load）**，而紧跟其后的下一条指令**需要用到这个刚读出来的数据（Use）**
> - **典型指令序列：**
>	I1: load  R1, 0(R2)   # 从内存读数据到 R1 (Load)
>	I2: add   R3, R1, R4   # 紧接着使用 R1 进行计算 (Use)
> 2. 为什么“转发（旁路）技术”对它无法完全消除停顿？
>	- 通常的 ALU 指令冒险可以通过转发技术实现“零停顿”，但 Load-Use 冒险做不到，原因在于数据的“**出生**”与“**急需**”时间点冲突：
>	- **数据何时出生？** 
>		- I1 是 Load 指令，数据在 **MEM（访存）阶段结束** 时才真正从内存中读出
>	- **数据何时急需？** 
>		- I2 在不卡顿的情况下，其 **EX（执行）阶段** 与 I1 的 MEM 阶段在**同一个时钟周期**进行
>		- ALU 在 EX 阶段开始时就需要输入 R1 的值
>	- **核心矛盾：** 转发技术无法将一个“还没有读出来（未出生）”的数据转发给正在执行的 ALU
> 3. 解决办法：硬件阻塞 + 转发
>	面对 Load-Use 冒险，处理器的基本策略是：**“硬等一个周期，然后再转发”**
> ##### 流水线时空图对比：
> **【错误/理想但无法实现的情况：不阻塞直接转发】**
> - I1 的 MEM 阶段和 I2 的 EX 阶段强行挤在周期 4，数据还没读出来，无法转发
>
> **【正确情况：硬件阻塞（Stall）1 个周期 + 转发】**
> 
| **指令**        | **1** | **2** | **3** | **4**   | **5**  | **6** | **7** |
| ------------- | ----- | ----- | ----- | ------- | ------ | ----- | ----- |
| **I1 (Load)** | IF    | ID    | EX    | **MEM** | WB     |       |       |
| **I2 (Use)**  |       | IF    | ID    | **阻塞**  | **EX** | MEM   | WB    |
>
>- **硬件介入：** 当译码器（ID 阶段）检测到 I2 要用的寄存器是 I1（Load 指令）的目的寄存器时，硬件触发阻塞
>- **错开时钟：** I2 在周期 4 保持**阻塞（Stall）**，其 EX 阶段被推迟到周期 5
>- **完美转发：** 在周期 4 结束时，I1 的 MEM 阶段完成，数据到手。在周期 5，硬件通过**转发（旁路）机制**，将数据直接送入 I2 的 EX 阶段

### 超流水线与超标量 
| 类型    | 本质         | 提高方式   |
| ----- | ---------- | ------ |
| 超流水线  | 增加流水段数     | 提高时钟频率 |
| 超标量   | 一个周期发射多条指令 | 提高并行度  |
| 普通流水线 | 一个周期发射一条指令 | ——     |
#### 超流水线
![[Pasted image 20260712211614.png]]
1. 在 **一个时钟周期** 内 再分段 （ 3 段）
2. 在一个时钟周期内一个功能部件使用多次（ 3 次）
3. 不能调整 指令的 执行顺序
4. 靠编译程序解决优化问题
#### 超标量流水线
![[Pasted image 20260712211509.png]]
1. 每个时钟周期内可 并发多条独立指令
2. 要配置多个功能部件
3. 不能调整 **指令的** 执行顺序
4. 通过编译优化技术，把可并行执行的指令搭配起来
#### 超长指令字技术
![[Pasted image 20260712211846.png]]
1. 由 **编译程序挖掘** 出指令间潜在的 **并行**性
2. 将 **多条** 能 **并行操作** 的指令组合成 一条
3. 具有 **多个操作码字段** 的 **超长指令字**（可达几百位）
4. 采用 **多个处理部件**
5. 主要改变 执行（EX） 部分

### 几个容易混淆的概念
| 名词             | 含义            |
| -------------- | ------------- |
| Bubble（气泡）     | 插入空周期         |
| Stall（停顿）      | 暂停流水线         |
| Forwarding（旁路） | 数据不经过 WB，直接转发 |
| Flush（冲刷）      | 清除错误进入流水线的指令  |
| Hazard（冒险）     | 流水线冲突         |
