白盒测试（White-box Testing）是一种**基于程序内部结构和逻辑**的测试方法。🌟简单说，就是**“打开盒子看里面”**——测试人员知道代码是怎么写的，然后根据代码结构来设计测试用例。
### 一句话总结
> 白盒测试 = “懂源码的人”用逻辑和路径去验证程序有没有按照预期运行。
---
### 和黑盒测试的区别

|对比点|白盒测试|黑盒测试|
|:--|:--|:--|
|关注点|程序**内部逻辑**与代码结构|程序**外部功能**与需求|
|测试依据|源代码、算法、控制流程|需求规格说明书|
|测试人员|需要懂代码的开发/测试工程师|不需要了解源码的测试人员|
|常用技术|语句覆盖、分支覆盖、路径覆盖等|等价类划分、边界值分析等|
|类比|拆开机器检查线路|按按钮看机器反应对不对|

---
### 白盒测试常见方法
1. **语句覆盖（Statement Coverage）**  
	 每一行代码至少执行一次。  
	 确保没有遗漏任何一行。
2. **分支覆盖（Branch Coverage）**  
    每个 if/else、case 等分支都执行到。  
    检查所有逻辑分支是否正确。
3. **条件覆盖（Condition Coverage）**  
    每个逻辑条件的“真/假”都至少执行一次。
4. **路径覆盖（Path Coverage）**  
    程序中所有可能的执行路径都被测试到。  
    最全面，但复杂度高。
5. **循环覆盖（Loop Coverage）**  
    针对循环语句测试 0 次、1 次、多次的情况。
---
### 举个例子：
假设我们有一段代码👇
```java
int max(int a, int b) {
    if (a > b)
        return a;
    else
        return b;
}
```
- **语句覆盖**：至少让 `if` 和 `else` 各执行一次
    - 用例1：a=5, b=3（走 if）
    - 用例2：a=2, b=7（走 else）
- **分支覆盖**：同上，两条分支都被覆盖
---
### 白盒测试的目的
- 检查程序逻辑正确性
- 提高代码覆盖率
- 提前发现潜在缺陷（如死循环、不可达代码、隐藏bug）
- 为代码优化提供依据
