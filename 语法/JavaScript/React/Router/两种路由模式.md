首先，**history模式** 和 **hash模式**，一般是说**前端路由（比如React Router、Vue Router）**在控制 URL 地址栏变化时，用的两种不同的策略。

---
### 1. hash模式（#号模式）
举个例子，地址长这样：
```
https://www.example.com/#/about
```
- `#` 后面的是前端自己控制的，浏览器不会把 `#/about` 发给服务器。
- 刷新页面时，服务器只收到 `https://www.example.com/`，不会因为 `#/about` 去找一个 `/about` 页面。
- **特点：**
	- 不需要服务器支持，直接前端就能搞定。
	- 兼容性最好（老浏览器也支持）。
	- URL中有个丑丑的 `#`。
- **原理：** 监听 `window.onhashchange` 事件，hash变了，前端代码就加载对应的内容。
**适合：** 静态托管、没法配服务器的小项目。
---
### 2. history模式（HTML5 History API）
举个例子，地址长这样：
```
https://www.example.com/about
```
- 没有 `#`，URL看起来很正常。
- 前端通过 `history.pushState()` 或 `history.replaceState()` 修改地址栏，但**不会刷新页面**。
- **特点：**
	- URL好看，没有`#`。
	- 可以使用浏览器的前进/后退。
	- 刷新页面时，服务器需要正确处理 `/about` 这个路径，不然会404。
- **原理：** 用 `pushState`、`popstate` 这些API操作浏览器历史记录和地址栏。
**需要配服务器：**
- 比如 nginx、Apache，要设置所有路由都返回 `index.html`，让前端接管。
否则比如你直接访问 `/about`，服务器找不到 `/about` 文件，就会报错。
---
### 总结一张小表：

|        |           |                                   |
| ------ | --------- | --------------------------------- |
|        | hash模式    | history模式                         |
| URL    | 有 `#`     | 没 `#`                             |
| 原理     | 监听hash变化  | History API (`pushState`<br><br>) |
| 兼容性    | 超好        | 需要HTML5支持                         |
| 刷新时服务器 | 不需要特别配置   | 需要配置返回`index.html`                |
| 使用场景   | 静态网站、简单应用 | 需要漂亮URL的SPA应用                     |
